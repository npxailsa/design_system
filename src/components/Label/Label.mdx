# Label

The `Label` component is a core building block of the design system, used to display text with optional icons and dropdown split functionality. It is designed to be highly systematic, with 120 possible variants based on size, weight, alignment, and state.

## Design Tokens

We use the following tokens to render our labels:

### Core Tokens

| Token Name | Description |
| :--- | :--- |
| `--label-text-color` | Primary text color |
| `--label-font-family` | Font family |
| `--label-icon-color` | Primary icon color |
| `--label-divider-color` | Dropdown divider color |
| `--label-letter-spacing` | Global letter spacing |

## Principles

Follow these common design principles by using labels in a cohesive, useful, and legible way.

**Design for universal understanding**
Labels should be clear, concise, and clearly describe the content or action they relate to.

**Balance simplicity and detail to create legibility**
Craft labels that use short, simple words, whilst containing enough context to properly identify the associated action or content.

**Maintain visual harmony**
Ensure labels work together as a cohesive system by adhering to consistent size, shape, and alignment guidelines across the entire interface.

**Use labels intentionally**
Labels provide crucial context and hierarchy to UI elements. Avoid unnecessary labels that repeat information clearly implied by icons or surrounding context.

### Do

Use systematic weights (bold, medium, light, thin) for consistent hierarchy.

### Don't

Overwrite internal component styles directly; use the token system. Avoid creating new custom weights or hardcoding pixel sizes.

## Contribution

When proposing updates or new properties for the Label component, consider the following:
1. Does the proposed change align with existing typography scales?
2. Does it maintain backward compatibility with our systematic sizing approach?

To contribute a new variation, please bring it to our Biweekly Design System meeting for review with the whole team.
